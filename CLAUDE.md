# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workspace layout

Two-crate cargo workspace:

- `libgssapi-sys/` — raw FFI bindings generated by `bindgen` at build
  time. There is no checked-in `bindings.rs`; `build.rs` writes it into
  `OUT_DIR` each build.
- `libgssapi/` — the safe wrapper. This is the public API.

## Common commands

```sh
cargo build                                    # no features (default is empty)
cargo build --all-features                     # iov + s4u + localname + store (MIT only)
cargo build --features iov,localname,store     # everything except s4u (MIT/Heimdal)
cargo run --example krb5 -- nfs@host.example.com
```

For tests, use the driver script — it sets up an in-process KDC and
runs the full suite. MIT runs natively against the host's krb5
install; Heimdal runs inside a podman container with target-heimdal/
as a separate build directory so it doesn't clobber the host build.
Apple GSS.framework runs natively on macOS against the system Heimdal
KDC.

```sh
tests/test.sh                # host default: Apple on macOS, else MIT
tests/test.sh mit            # MIT (host)
tests/test.sh apple          # Apple GSS.framework (host, macOS only)
tests/test.sh heimdal        # Heimdal in container
tests/test.sh all            # both Linux impls (MIT + Heimdal)
```

The Heimdal container image (`tests/docker/Dockerfile.heimdal`) is
built on first run and cached. Tests pass identically across all three
impls — the `TestKdc` fixture in `tests/common/mod.rs` detects which
KDC binaries are present and branches on MIT vs Heimdal command syntax.
On macOS the Heimdal KDC ships as a launchd helper inside
`Heimdal.framework`; the fixture finds it there and adds
`--listen-on-network --no-sandbox` so it binds a TCP/UDP port standalone.
The Apple run uses the default feature set rather than `--all-features`
because `s4u` is MIT-only (see below).

Pure-Rust tests also run under Miri:

```sh
cargo +nightly miri test -p libgssapi --no-default-features pure_rust_tests
```

The examples in `libgssapi/examples/` are smoke tests against a real
KDC; rarely needed now that `tests/krb5_mutual_auth.rs` does the same
thing as a `#[test]`. When MIT Kerberos returns its usual unhelpful
"Unspecified GSS failure" during debugging, set
`KRB5_TRACE=/dev/stderr` to see what actually went wrong.

A `rustfmt.toml` exists, but per the user's global rules **don't run
`cargo fmt`** — it produces huge unrelated diffs.

## Build-time GSSAPI selection (`libgssapi-sys/build.rs`)

There are three supported underlying implementations, chosen at build
time:

| Detected via | Implementation | Header used | Link line |
|---|---|---|---|
| `pkg-config mit-krb5-gssapi` (or finds `libgssapi_krb5.so*`) | MIT | `src/wrapper_mit.h` | `-lgssapi_krb5` |
| `pkg-config heimdal-gssapi` (or finds `libgssapi.so*`) | Heimdal | `src/wrapper_heimdal.h` | `-lgssapi` |
| `target_os = "macos"` | Apple GSS.framework | `src/wrapper_apple.h` | `-framework GSS` |

Windows panics — use SSPI. When cross-compiling (`HOST != TARGET`),
`pkg-config` is skipped and the library-search path runs. Nix's
`NIX_CFLAGS_COMPILE` is forwarded to bindgen so it can find headers in
the Nix store.

Two env-var overrides (both `rerun-if-env-changed`):

- `LIBGSSAPI_IMPL` = `mit` | `heimdal` | `apple` forces the
  implementation, bypassing all autodetection. The escape hatch for
  machines with both stacks installed.
- `LIBGSSAPI_PREFIX` = colon-separated install prefixes. Each adds
  `<prefix>/include` to bindgen (`-I`) and `<prefix>/lib` to the linker
  (`-L`), and is searched for the library during autodetection. For
  source/custom installs that pkg-config can't see.

The library search (`which`) only runs when pkg-config is unavailable.
It checks the `LIBGSSAPI_PREFIX` lib dirs, the `krb5-config --prefix`
lib dir, `LD_LIBRARY_PATH`, and the standard system lib dirs
**non-recursively** (it used to shell out to a recursive `find`; that
was dropped). Non-recursive means it won't see multiarch subdirs like
`/usr/lib/x86_64-linux-gnu` — but those distros ship pkg-config files,
so the search never runs there anyway. If autodetection finds nothing it
panics pointing at `LIBGSSAPI_IMPL` / `LIBGSSAPI_PREFIX`.

`try_pkgconfig` probes `mit-krb5-gssapi` before `heimdal-gssapi` and
takes the first hit, so on a machine with **both** dev stacks installed
the build picks MIT (use `LIBGSSAPI_IMPL` to override). This bit the
Heimdal test container: `rust:bookworm` ships `krb5-multidev` (MIT's
`.pc`/headers) via buildpack-deps, so the container silently built MIT
until `Dockerfile.heimdal` started purging that package. If a "Heimdal"
build links `-lgssapi_krb5`, this is why.

Important consequence for Heimdal: the struct *tag* names bindgen emits
differ between impls (`gss_name_struct` on MIT vs
`gss_name_t_desc_struct` on Heimdal). The safe wrapper must never name
those tags — only the portable handle typedefs (`gss_name_t`,
`gss_cred_id_t`, …). To make a null handle, write
`let mut x: gss_name_t = ptr::null_mut();`, not
`ptr::null_mut::<gss_name_struct>()`.

Both wrappers include `consts.h`, which re-exports a bunch of
`GSS_C_*`/`GSS_S_*` macros as proper `OM_uint32` constants with
underscore-prefixed names (`_GSS_S_FAILURE` etc.). The safe wrapper
imports these `_`-prefixed names because bindgen can't see the macros
themselves.

## Feature flags

All declared in `libgssapi/Cargo.toml`. **The default is empty.** None of
the optional features build on every implementation, so the only honest
cross-platform default is no features — a feature does exactly what it
says, and enabling one the linked impl can't provide is a *compile error*,
not a silent no-op. `all` enables everything (MIT only).

Per-impl support:

- `iov` — `wrap_iov`/`unwrap_iov`/`wrap_iov_length` and the `GssIov` /
  `GssIovFake` types. Gated `#[cfg(feature = "iov")]` throughout
  `context.rs`, with the types in a submodule in `util.rs`. **MIT +
  Heimdal** — Apple's GSS.framework has no `gss_wrap_iov`.
- `s4u` — Kerberos S4U (constrained delegation): `Cred::impersonate`,
  `Cred::store_into`, `Cred::proxy` lookup via
  `GSS_KRB5_GET_CRED_IMPERSONATOR`, and the internal `BufSet` type.
  **MIT only** — Heimdal and Apple provide neither
  `gss_acquire_cred_impersonate_name` nor `gss_store_cred_into`.
- `localname` — `Name::local_name` (POSIX local-name mapping). **MIT +
  Heimdal** — Apple has no `gss_localname`.
- `store` — `Cred::store` (writes to default ccache via the older
  `gss_store_cred`). **MIT + Heimdal** — Apple has no `gss_store_cred`.

Because the default is empty, the test runs select features explicitly:
the MIT host run uses `--all-features`; the Heimdal container run uses
`--features iov,localname,store` (everything but s4u); the Apple run uses
the default (nothing — its full supported surface). The `krb5_iov` example
declares `required-features = ["iov"]` so it's skipped unless iov is on.

Consumers that want a feature only where it's available should enable it
in a target-specific dependency table rather than unconditionally, e.g.:

```toml
[target.'cfg(not(target_os = "macos"))'.dependencies]
libgssapi = { version = "...", features = ["iov", "localname", "store"] }
```

When adding code that calls an FFI function only present on some
implementations, gate it behind the appropriate feature and update both
the safe wrapper and any necessary imports inside `cfg` blocks. Don't try
to make a feature a platform-conditional no-op — let the missing function
surface as an honest build error on impls that lack it.

## Safe-wrapper architecture

The wrapper is a thin RAII layer over GSSAPI's C handle types. Each
owned handle is a `repr(transparent)` newtype around the C pointer and
implements `Drop` to call the matching `gss_release_*`. Manual `Send`
/ `Sync` impls are everywhere because the raw pointers don't auto-derive
them — these are sound as long as the underlying mech is thread-safe,
which both MIT and Heimdal are.

The handle types and their lifecycles:

- **`Buf`** (`util.rs`) — owned output buffer GSSAPI allocated.
  `Drop` calls `gss_release_buffer`. Derefs to `[u8]`.
- **`BufRef<'a>`** — borrowed input buffer. Never released, lifetime
  tied to the borrowed slice. **Only pass to GSSAPI calls that
  don't mutate input** — the comment on the type spells out why.
- **`Cred`** — `Arc<CredInner>`, so credentials are cheaply clonable
  and shared. The `Arc` is what gates the `gss_release_cred` call.
- **`Name`** — owns a `gss_name_t`. `Debug`/`Display` call
  `display_name()`.
- **`Oid`** — `repr(transparent)` over `gss_OID_desc`. Many static
  `GSS_*` OIDs are defined in `oid.rs` with their BER encodings hard
  coded (the standard requires OIDs to live in static memory). The
  `OIDS` lazy-static map provides nice `Debug` output. Equality is by
  byte comparison, not pointer.
- **`OidSet`** — owned `gss_OID_set`.
- **`ClientCtx`**, **`ServerCtx`** — wrap `gss_ctx_id_t` plus a small
  state-machine enum (`Uninitialized` → `Partial` → `Complete`, or
  `Failed`). Both implement the `SecurityContext` trait, which is the
  unified post-handshake API (`wrap`/`unwrap`/`get_mic`/`verify_mic`/
  info getters).

### Context handshake pattern

`ClientCtx::step(tok, channel_bindings)` and
`ServerCtx::step(tok)` both return `Result<Option<Buf>, Error>`:

- `Ok(Some(tok))` — keep going, send this token to the peer.
- `Ok(None)` — handshake complete from our side.
- `Err(_)` — fatal; the state machine moves to `Failed` and further
  `step()` calls return the same error.

Drive both with a loop until `Ok(None)`. See
`libgssapi/examples/krb5.rs` for the canonical shape.

### Info-getter dispatch

Both `Cred` and the contexts have `info()`, plus per-field getters
(`source_name`, `lifetime`, `mechanism`, etc.). Internally these all
go through one `info_c(...)` call that takes a `CtxInfoC` /
`CredInfoC` selector struct — `Some(ptr::null_mut())` means "fill this
field", `None` means "skip". The full `info()` requests everything;
single-field getters request just one. When adding a new info field,
add it to the C selector struct, the FFI dispatch, and both the
fine-grained and full-info wrappers.

### Error handling

`Error { major: MajorFlags, minor: u32 }`. `MajorFlags` is a
`bitflags!` mirror of the routine/calling/supplementary error
constants. `Display` formats both halves by calling
`gss_display_status` repeatedly until `message_context == 0`. The
`gss_error()` helper masks off supplementary bits and is what most FFI
callsites use to decide "fatal vs continue-needed".

## Conventions worth following

- New FFI calls go through the existing `unsafe fn` helpers in each
  module rather than being inlined into trait methods — keeps the
  `unsafe` blocks small and lets `ClientCtx` and `ServerCtx` share
  implementations.
- When a C-side call partially succeeds before failing (allocates
  some output then errors), free what was allocated before returning
  `Err` — `context::info` and `credential::info_c` show the pattern.
- For new credential/name/buffer types crossing the FFI boundary,
  mirror the existing `to_c` / `from_c` `pub(crate) unsafe` pair.
